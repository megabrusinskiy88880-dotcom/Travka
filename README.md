# Бедрок-клад — магазин кладов Minecraft

Простой сайт для приёма заявок на "клады" в Minecraft. Без онлайн-оплаты — заявки подтверждаются вручную через админку, координаты клада ты вписываешь сам после подтверждения.

## Как это устроено

- **Главная страница** — вход по нику (без пароля). Если ника нет в базе, профиль создаётся автоматически.
- **/shop** — лавка с лотами (видна только вошедшим по нику).
- **/cabinet** — личный кабинет: список заявок, статус, координаты клада после подтверждения.
- **Секретная админка** — доступна по пути `/ADMIN_PATH` (задаётся в переменных окружения), вход по паролю. В обычной навигации сайта ссылки на неё нет — её знаешь только ты.
- **База данных** — простой JSON-файл (`data/db.json`), без сервера БД.

## Админка умеет

1. **Заявки** — менять статус (ожидает / подтверждено / выдано / отменено), вписывать координаты клада, добавлять заметку для покупателя. Как только статус меняешь на "подтверждено" или "выдано", товар списывается со склада.
2. **Лоты** — создание, редактирование, удаление, загрузка фото (автоматически сжимается), скрытие лота из лавки без удаления.
3. **Пользователи** — список всех, кто заходил по нику.

## ⚠️ Важно про Railway: постоянное хранилище (Volume)

Railway по умолчанию **стирает файловую систему при каждом деплое**. Это значит, что без дополнительной настройки твоя база `data/db.json` и загруженные фото лотов будут пропадать при каждом обновлении кода.

**Чтобы данные сохранялись, обязательно подключи Railway Volume:**

1. В проекте на Railway: правый клик по сервису → **Attach Volume** (или через Command Palette ⌘K → "New Volume").
2. Смонтируй volume на путь `/app/data` — здесь будет храниться `db.json`.
3. Создай **второй volume** (или расширь на общий раздел, если тариф позволяет) и смонтируй на `/app/public/uploads` — здесь хранятся фото лотов.
   - Если план Railway позволяет только один volume на сервис, смонтируй его на `/app/data` и **вынеси фото во внешнее хранилище** (см. ниже) — либо храни фото прямо в volume `/app/storage` и укажи `UPLOAD_DIR=/app/storage/uploads`, `DB_PATH=/app/storage/data/db.json` — тогда обе директории попадут в один volume.
4. Убедись, что переменные окружения указывают на эти пути (см. ниже).

Проще всего: один volume, смонтированный на `/app/storage`, и такие переменные:

```
DB_PATH=/app/storage/data/db.json
UPLOAD_DIR=/app/storage/uploads
```

Без этого шага всё будет работать, но **любой новый деплой обнулит все лоты, заявки и фото**.

## Переменные окружения (Railway → Variables)

| Переменная | Обязательна | Описание |
|---|---|---|
| `ADMIN_PATH` | Да | Секретный путь до админки, например `admin-9f2k7x-mine`. Никому не сообщай. |
| `ADMIN_PASSWORD` | Да | Пароль для входа в админку. |
| `SESSION_SECRET` | Да | Любая длинная случайная строка для подписи сессий. |
| `DB_PATH` | Нет (но нужна для volume) | Путь к файлу базы данных. По умолчанию `./data/db.json`. |
| `UPLOAD_DIR` | Нет (но нужна для volume) | Путь к папке с фото лотов. По умолчанию `./public/uploads`. |
| `NODE_ENV` | Рекомендуется | Установи `production`. |
| `PORT` | Нет | Railway подставляет сама. |

Задать их можно на вкладке **Variables** твоего сервиса в Railway.

## Локальный запуск

```bash
npm install
cp .env.example .env
# отредактируй .env — впиши свой ADMIN_PATH, ADMIN_PASSWORD, SESSION_SECRET
node server.js
```

Открой `http://localhost:3000` — попадёшь на вход по нику.
Админка будет на `http://localhost:3000/<твой ADMIN_PATH>`.

## Деплой на Railway

1. Залей этот проект в свой GitHub-репозиторий (или используй `railway up` из CLI без гита).
2. В Railway: **New Project → Deploy from GitHub repo**, выбери репозиторий.
3. Railway сам определит Node.js-проект и установит зависимости (Nixpacks).
4. Зайди в **Variables** и добавь все переменные из таблицы выше.
5. Подключи **Volume** (см. раздел выше) — это критично, не пропускай.
6. Дождись деплоя, открой публичный URL, который выдаст Railway.
7. Проверь: главная страница → вход по нику → лавка (пустая, лотов ещё нет) → зайди в `/<ADMIN_PATH>` → войди паролем → создай первый лот.

## Как пользоваться каждый день

1. Покупатель заходит на сайт, вводит ник, видит лоты, жмёт "Оформить заявку".
2. Ты заходишь в админку → вкладка **Заявки** → видишь новую заявку со статусом "Ожидает".
3. Договариваешься с покупателем (в игре/дискорде/где угодно — оплата вне сайта).
4. Прячешь клад в игре, копируешь координаты.
5. В админке вписываешь координаты в поле "Координаты клада", меняешь статус на **Подтверждено** (или сразу **Выдано**), жмёшь "Сохранить".
6. Покупатель заходит в свой **Кабинет** и видит координаты.

## Структура проекта

```
server.js              — точка входа, роуты страниц, статика
db.js                   — чтение/запись JSON-базы
middleware/auth.js      — проверка сессии пользователя и админа
routes/user.js          — API: вход по нику, лоты, заявки
routes/admin.js         — API: логин админа, CRUD лотов, управление заявками
public/index.html       — вход по нику
public/shop.html        — лавка с лотами
public/cabinet.html     — личный кабинет с заявками
public/admin.html       — админ-панель (вход по паролю, вкладки: заявки/лоты/пользователи)
public/style.css        — общие стили
data/db.json            — база данных (лоты, заявки, пользователи)
public/uploads/         — загруженные фото лотов
```

## Безопасность — на что обратить внимание

- Секретность админки держится на двух вещах: неизвестности пути `ADMIN_PATH` и пароле `ADMIN_PASSWORD`. Обязательно смени оба значения по умолчанию перед запуском в проде.
- `SESSION_SECRET` тоже смени — от него зависит, можно ли подделать сессионные cookie.
- Не публикуй `.env` и не коммить его в git (он уже в `.gitignore`).
- Ссылки на `/ADMIN_PATH` нигде в публичном интерфейсе не размещены, но сам путь всё равно передаётся по сети открытым текстом при заходе — используй Railway-домен с HTTPS (Railway включает его по умолчанию), не давай ссылку на админку в открытых чатах.
